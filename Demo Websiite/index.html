Get Started
Install NSB and prepare your environment. Estimated time: 10 minutes.
✓
Check prerequisites
You need Python 3.8+, Git, and Homebrew installed before proceeding.
bash
$ python3 --version    # must be 3.8 or higher
$ git --version
$ brew --version
2
Install NSB server
Install the NSB message server via Homebrew. This handles routing between your simulator and client processes.
bash
$ brew tap ucsc-ospo/nsb
$ brew install nsb-server
$ nsb-server --version    # verify: NSB Server 0.4.2
You should see: NSB Server 0.4.2. If you see "command not found", restart your terminal.
3
Install Python client library
Install the Python package that lets your code communicate with the NSB server.
bash
$ python3 -m venv nsb-env
$ source nsb-env/bin/activate
$ pip install nsb-client
You should see: Successfully installed nsb-client-0.4.2
4
Clone the examples repository
The examples repo contains the basic_client.py you'll use in the quickstart.
bash
$ git clone https://github.com/ucsc-ospo/nsb-examples
$ cd nsb-examples
Next: Run your first simulation →

Quickstart
Run your first simulation. No prior NSB knowledge required.
~5 minutes
1
Create your config file

Copy this exactly. You don't need to understand every field yet — that's what the Config Reference is for.

config.yaml
simulator_mode: 1
server_port: 5000
endpoints:
  - name: client1
    port: 5001
  - name: client2
    port: 5002
Save this as config.yaml in your current directory.
2
Start the NSB server

Open a new terminal window and run:

terminal 1
$ nsb-server --config config.yaml
You should see: NSB Server listening on port 5000. Leave this terminal open.
⚠ Port 5000 in use? Change server_port in config.yaml. See port conflicts →
3
Run the example client

In a second terminal window, inside nsb-examples/:

terminal 2
$ source nsb-env/bin/activate
$ python examples/basic_client.py
4
Watch the simulation run

You should see output like this in your terminal:

expected output
▶  Connecting to NSB server at localhost:5000
✓  client1 connected successfully (port 5001)
✓  client2 connected successfully (port 5002)
▶  Sending message: Hello from client1 → client2
▶  Sending message: Hello from client2 → client1
🎉  Simulation complete. 10 messages exchanged in 0.34s
🎉

Docs / Architecture
System Architecture
Last updated: March 2026 · Edit this page

NSB is built around a central message server that acts as a bridge between simulator processes and client controllers. Understanding this model is the key to using NSB effectively.

Core Components

NSB has three main components that work together:

  • NSB Server — the central message routing process. Started once per simulation session.
  • Client endpoints — your Python or C++ programs that connect to the server and exchange messages.
  • Simulator bridge — the adapter that connects your simulator (ns-3, OMNeT++) to the NSB message bus.

Config Reference

The config.yaml file controls all runtime behaviour. Every field is documented below.

FieldTypeDefaultDescription
simulator_mode required — Set to 1 for basic mode, 2 for ns-3 bridge, 3 for OMNeT++.
server_port optional 5000 Port the NSB server listens on. Change if 5000 is already in use.
endpoints required — List of client endpoints. Each needs a unique name and port.
endpoints[].name required — Unique identifier for this endpoint. Used in message routing.
endpoints[].port required — Port this client endpoint listens on. Must not conflict with other endpoints.
log_level optional INFO Logging verbosity: DEBUG, INFO, WARNING, ERROR.

Message Flow

When a client sends a message, the NSB server receives it, looks up the target endpoint by name, and forwards it on the correct port. Clients never communicate directly — all traffic routes through the server.

python — sending a message
from nsb_client import NSBClient

client = NSBClient("client1", server_port=5000)
client.connect()

# Send a message to client2
client.send(to="client2", payload={"type": "PING", "seq": 1})

# Receive a reply
msg = client.recv(timeout=2.0)
print(msg.payload)

About NSB

Origin, goals, and the team behind the project.

Network Simulation Bridge was created at UC Santa Cruz as part of the Open Source Program Office (UCSC OSPO). The goal is to lower the barrier for researchers and students who want to use mature network simulators without dealing with complex integration plumbing.

NSB is actively developed and welcomes contributors at all levels. The GSoC 2026 program is bringing major improvements to documentation, onboarding, and the website.
